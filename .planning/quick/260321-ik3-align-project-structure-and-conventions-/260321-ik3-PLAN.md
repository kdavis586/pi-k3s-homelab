Align both Helm charts (jellyfin, pihole) and Flux HelmReleases with industry-standard conventions for GitOps K3s homelabs.

Aligning with conventions makes the charts consistent with what the Helm and Flux communities expect, improves `kubectl` filtering, and makes adding future charts easier by establishing a template pattern.

  <action>
Create `_helpers.tpl` for each chart following standard Helm scaffold conventions. Each _helpers.tpl should define these named templates:

1. `{chartname}.name` - the chart name (trimmed to 63 chars)
2. `{chartname}.fullname` - release-qualified name (trimmed to 63 chars). Since these are single-instance homelab charts, fullname can just return the chart name for simplicity (avoids renaming all resources).
3. `{chartname}.labels` - standard label block producing:
   - `app.kubernetes.io/name: {{ include "{chartname}.name" . }}`
   - `app.kubernetes.io/instance: {{ .Release.Name }}`
   - `app.kubernetes.io/version: {{ .Chart.AppVersion | quote }}`
   - `app.kubernetes.io/managed-by: {{ .Release.Service }}`
   - `helm.sh/chart: {{ include "{chartname}.chart" . }}`
4. `{chartname}.selectorLabels` - just name + instance (subset used in matchLabels and pod template)
5. `{chartname}.chart` - chart name + version string

For jellyfin, {chartname} = "jellyfin". For pihole, {chartname} = "pihole".

Create `.helmignore` for each chart with standard entries: `.git`, `.gitignore`, `*.swp`, `.DS_Store`, `# comment explaining purpose`.

IMPORTANT: The `selectorLabels` helper replaces the existing `app: jellyfin` / `app: pihole` selectors. This means existing pods will NOT match the new selectors. This is acceptable for a homelab -- Flux will do a Recreate deploy. But the old bare `app:` label should NOT be included in selectorLabels to keep it clean.

  <action>
**For every template in both charts**, replace hardcoded labels with helper calls:

In metadata sections, replace:
```yaml
labels:
  app: jellyfin  # or pihole
```
with:
```yaml
labels:
  {{- include "jellyfin.labels" . | nindent 4 }}
```
(Use nindent 4 for top-level metadata, nindent 8 for nested pod template metadata.)

In `spec.selector.matchLabels` and `spec.template.metadata.labels` of Deployments, use `selectorLabels` (NOT full labels -- selectorLabels is immutable after creation):
```yaml
selector:
  matchLabels:
    {{- include "jellyfin.selectorLabels" . | nindent 6 }}
template:
  metadata:
    labels:
      {{- include "jellyfin.selectorLabels" . | nindent 8 }}
```

Keep all existing functionality intact -- the hardcoded `namespace: jellyfin` and `namespace: pihole` on each resource MUST remain (per project decision: hardcode namespace, not Release.Namespace). Only labels change.

Keep all existing comments explaining WHY things are configured a certain way (e.g., Recreate strategy comment, hostNetwork comment, initContainer comment). Do not remove any comments.

**Delete namespace.yaml** from both `charts/jellyfin/templates/` and `charts/pihole/templates/`.

**Update Flux HelmReleases** (`flux/apps/jellyfin.yaml` and `flux/apps/pihole.yaml`):
- Change `createNamespace: false` to `createNamespace: true`

This moves namespace lifecycle management from the chart to Flux, which is the standard Flux convention.
