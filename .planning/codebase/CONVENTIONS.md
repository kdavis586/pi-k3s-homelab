# Coding Conventions

**Analysis Date:** 2026-03-18

## Overview

This is an Infrastructure-as-Code (IaC) project using Ansible, Kubernetes manifests, Jinja2 templates, and Makefiles. All configuration is declarative and derived from a single source of truth. There are no application-level unit tests; validation occurs through Ansible idempotency and kubectl deployments.

## Naming Patterns

### Files

**YAML Files:**
- K8s manifests: `lowercase-name.yaml`
  - Examples: `deployment.yaml`, `service-web.yaml`, `00-namespace.yaml` (00 prefix for apply order)
  - Namespace manifests use `00-` prefix to ensure they're created first

**Ansible Files:**
- Playbooks: `kebab-case-name.yaml` in `ansible/playbooks/`
- Variables: `all.yaml` in `ansible/group_vars/`
- Generated files get descriptive comments: `# Generated by: make generate`

**Jinja2 Templates:**
- `.j2` extension
- Examples: `user-data.j2`, `network-config.j2`, `jellyfin-deployment.j2`
- Generated files override with target name: template `jellyfin-deployment.j2` → `k8s/jellyfin/deployment.yaml`

**Shell/Make:**
- Makefile targets: `lowercase-with-dashes`
- Help text uses `##` comments: `target: ## Description`

### Variables

**Ansible:**
- `snake_case` for all variables: `server_hostname`, `usb_device`, `dns_servers`
- Global vars in `ansible/group_vars/all.yaml`
- Loop variables use item: `item.hostname`, `item.ip`

**Jinja2 Templates:**
- Access Ansible variables directly: `{{ variable_name }}`
- Conditionals: `{% if condition %} ... {% endif %}`
- Loops: `{% for item in list %} ... {% endfor %}`

**Kubernetes:**
- Container names: `lowercase` (e.g., `jellyfin`, `pihole`)
- Service selectors: `app: lowercase-name`
- Port names: `lowercase-dash` (e.g., `dns-udp`, `dns-tcp`)

### Kubernetes Resources

**Labels and Selectors:**
- Standard label: `app: <resource-name>`
- Custom labels: `homelab/node-group: <group-name>` (domain-prefixed for cluster-local extensions)
- All resources include `app` label for service selection

**Resource Names:**
- Deployments: singular form, lowercase, dash-separated
  - Example: `jellyfin` (not `jellyfin-deployment`)
- Services: singular form + role suffix if multiple services per app
  - Example: `pihole-web` for the web UI service
- ConfigMaps: descriptive lowercase
  - Example: `local-path-config`
- Namespaces: singular, lowercase
  - Example: `jellyfin`, `pihole`
- PersistentVolumeClaims: `<app>-<component>`
  - Example: `jellyfin-config`

**Comments in Manifests:**
- Explain non-obvious decisions or constraints
- Examples:
  - "Recreate strategy: Jellyfin cannot run multiple instances sharing the same config volume"
  - "Pin to apple-pi — that's where the USB drive is mounted"
  - "hostNetwork required for DHCP — DHCP clients broadcast to 255.255.255.255:67"

## Code Style

### YAML Formatting

**Indentation:**
- 2 spaces (consistent across Ansible, K8s, cloud-init)
- No tabs

**Structure:**
- Start playbooks with `---`
- Start K8s manifests with `---`
- Comments above sections for clarity

**Order:**
- In Ansible tasks: `name` → module name → parameters
- In K8s specs: `metadata` → `spec` sections
- In container specs: `name` → `image` → `ports` → `env` → `resources` → `volumeMounts` → probes

### Comments

**When to Comment:**
- Explain *why*, not *what*
- Document non-obvious k8s configurations (e.g., hostNetwork, securityContext)
- Explain workarounds: "k3s iptables rules intercept loopback port 53..."
- Document hardware constraints or gotchas

**When NOT to Comment:**
- Self-explanatory Ansible tasks
- Standard resource fields
- Boilerplate code

**Examples of Good Comments:**
```yaml
# k3s iptables rules intercept loopback port 53 traffic, breaking systemd-resolved's
# stub listener at 127.0.0.53. Fix: disable the stub and point /etc/resolv.conf at
# the real upstream DNS IPs directly.

# Pin node DNS to external resolvers (ignore DHCP-assigned DNS) to prevent
# chicken-and-egg deadlock if Pi-hole pod restarts.

# hostNetwork required for DHCP — DHCP clients broadcast to 255.255.255.255:67
# which iptables cannot DNAT (unicast only).
```

## File Organization

### Ansible Playbooks

**Location:** `ansible/playbooks/`

**Convention:**
- One playbook per major operation: `base-setup.yaml`, `k3s-install.yaml`, `deploy.yaml`
- Play names are descriptive: `- name: Install K3s server`
- Multiple plays in one file are organized by function (e.g., server setup → agent setup → verification)
- Handlers at the bottom of plays

**Task Naming:**
- Descriptive present tense: `Ensure required packages are installed`
- State the intent, not the tool: not `Run apt install`, but `Ensure required packages are installed`

**Example Pattern:**
```yaml
---
# Function description
# Usage: ansible-playbook <path>

- name: <Primary function>
  hosts: <target>
  become: <true/false>
  tasks:
    - name: <Descriptive task name>
      ansible.<module>: ...

  handlers:
    - name: <Handler name>
      ansible.<module>: ...

- name: <Secondary play if grouped>
  hosts: <different target>
  ...
```

### Kubernetes Manifests

**Location:** `k8s/<app>/`

**Convention:**
- One file per resource kind: `deployment.yaml`, `service.yaml`, `ingress.yaml`, `pvc.yaml`
- Namespace definitions in `00-namespace.yaml` (numbered to ensure creation order)
- Generated manifests include: `# Generated by: make generate (source: <template>)`
- Comments explain constraints or non-standard configurations

**File Order in Directory:**
- `00-namespace.yaml` (applied first)
- `deployment.yaml` or `daemonset.yaml`
- `pvc.yaml` (if needed)
- `service.yaml`
- `ingress.yaml`

### Cloud-init and Templates

**Location:** `cloud-init/templates/` (Jinja2) → `cloud-init/` (generated per-node)

**Templates:**
- `user-data.j2` — OS config, SSH keys, package installation
- `network-config.j2` — Network interface configuration
- `meta-data.j2` — Instance ID for cloud-init tracking

**Variables from Source of Truth:**
- `ansible/group_vars/all.yaml` is rendered into all templates
- Variables are accessed as `{{ variable_name }}`

## Error Handling

**Ansible Patterns:**

- Use `changed_when: false` for read-only operations (stat checks, queries)
- Use `ignore_errors: true` for expected failures: `ignore_errors: true  # ufw may not be running`
- Use `when` clauses to check preconditions: `when: usb_dev.stat.exists`
- Register results and use them for conditionals: `register: usb_uuid` then `when: usb_uuid.stdout != ""`
- Use `until` + `retries` for polling: waiting for API servers or pod status

**Example:**
```yaml
- name: Wait for K3s API server to be healthy
  ansible.builtin.uri:
    url: "https://{{ server_ip }}:6443/healthz"
    validate_certs: false
    status_code: [200, 401]  # Accept both success codes
  register: k3s_health
  until: k3s_health.status in [200, 401]
  retries: 30
  delay: 5
```

**Kubernetes Patterns:**

- Resource probes (liveness + readiness) define health checks
- Deployment strategies: `Recreate` for single-instance (Jellyfin), `Rolling` for stateless
- Resource limits/requests: always specify both for scheduling
- Selectors and labels: use `app` label consistently for pod-service binding

## Logging

**Ansible:**
- Use `ansible.builtin.debug: msg:` for informational output
- Messages are multi-line where useful: `msg: |` with indented text
- Examples: kubeconfig merge instructions, USB drive status reminders

**Kubernetes:**
- Container logs viewed via: `kubectl logs -n <namespace> -l app=<app-name>`
- Make target for convenience: `logs:` tail Jellyfin logs

**No Application Logging:**
- This is purely infrastructure; application logging is handled by container images (Jellyfin, Pi-hole)
- Host-level logs via journalctl on each Pi

## Import Organization

**Ansible Task Organization:**
- Imports via `include` or `vars` at the top of plays
- Variables imported from `ansible/group_vars/all.yaml` automatically in all plays with `hosts: all`
- Playbooks reference templates with full relative paths: `src: "{{ playbook_dir }}/../../cloud-init/templates/user-data.j2"`

**Kubernetes Include Pattern:**
- No include statements; all manifests applied recursively: `kubectl apply -R -f k8s/`
- Namespace applied first via `00-namespace.yaml` prefix

## Function/Task Design

**Ansible Tasks:**
- One logical action per task
- Task names are specific: not `Configure system` but `Pin node DNS to external resolvers (ignore DHCP-assigned DNS)`
- Handler invocation via `notify:` — no manual restarts
- Loop variables: use `item` or descriptive names in `loop_var:`

**Kubernetes Deployment Spec:**
- Replica strategy clear: `Recreate` vs `Rolling`
- Resource requests + limits always set
- Probes include: `initialDelaySeconds`, `periodSeconds`, `failureThreshold`
- Selector labels match pod labels exactly

## Module Design

**Playbook Organization:**
- One playbook file per major workflow (setup, install, deploy, generate)
- Multiple related plays can be in one file if they're sequential dependencies
- Plays are logically ordered: prerequisites first (e.g., server install before agent install)

**Manifest Organization:**
- One directory per application: `k8s/jellyfin/`, `k8s/pihole/`
- Shared infrastructure in `k8s/storage/`, `k8s/flux-system/`
- All manifests in app directory are applied together

**Configuration Files:**
- Single source of truth: `ansible/group_vars/all.yaml`
- Run `make generate` to propagate to all targets
- Do NOT edit generated files directly; edit the source template and re-generate

## Port and Resource Conventions

**Container Port Naming:**
- DNS: `dns-udp`, `dns-tcp`
- HTTP/Web: `http` (implied TCP)
- DHCP: `dhcp` (UDP)

**Resource Requests/Limits:**
- Always set `requests` (scheduler needs them)
- Always set `limits` (prevent resource hogging)
- Memory: `Mi` (mebibytes), CPU: `m` (millicores) or whole numbers
- Examples: `memory: 512Mi`, `cpu: 250m`

**Node Affinity:**
- Use `nodeSelector` for simple hostname matching: `kubernetes.io/hostname: apple-pi`
- Use `affinity.nodeAffinity` for label-based selection: `homelab/node-group: workloads`
