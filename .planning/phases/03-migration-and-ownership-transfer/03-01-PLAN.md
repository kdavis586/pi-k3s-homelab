Validates the delete-then-reconcile approach before applying it to Pi-hole.

Output: Jellyfin running under Flux ownership with READY=True HelmRelease, PVC preserved.

This task has two stages: pre-flight assessment, then Jellyfin ownership transfer.

**Stage 1: Pre-flight cluster assessment**

Run these diagnostic commands to understand current cluster state (all use `--kubeconfig ~/.kube/config-pi-k3s`):

1. Check current HelmRelease status for BOTH workloads:
   ```
   flux get helmreleases -A --kubeconfig ~/.kube/config-pi-k3s
   ```
   Record whether jellyfin and pihole show READY=True, False, or error states.

2. Check for existing Helm release Secrets (determines if Helm will install or upgrade):
   ```
   kubectl get secrets -n jellyfin --kubeconfig ~/.kube/config-pi-k3s | grep helm.release
   kubectl get secrets -n pihole --kubeconfig ~/.kube/config-pi-k3s | grep helm.release
   ```

3. Check node labels (needed for Pi-hole in Plan 02, but assess now):
   ```
   kubectl get nodes --show-labels --kubeconfig ~/.kube/config-pi-k3s | grep homelab
   ```
   If `homelab/node-group=workloads` is NOT present on apple-pi and pumpkin-pi, note this for Plan 02.

4. Check if pihole-dns LoadBalancer Service exists on cluster:
   ```
   kubectl get svc -n pihole --kubeconfig ~/.kube/config-pi-k3s
   ```
   If a `pihole-dns` service exists, note it for Plan 02 (may need to be added to chart or explicitly deleted).

5. Check current Jellyfin resources and their field managers:
   ```
   kubectl get all,ingressroute,pvc -n jellyfin --kubeconfig ~/.kube/config-pi-k3s
   ```

6. Verify the Jellyfin PVC exists and note its spec:
   ```
   kubectl get pvc jellyfin-config -n jellyfin -o yaml --kubeconfig ~/.kube/config-pi-k3s
   ```

Record ALL findings. These inform the transfer approach.

**Stage 2: Jellyfin ownership transfer (delete-then-reconcile)**

Based on pre-flight results, execute the ownership transfer:

A. If a Helm release Secret exists for jellyfin (from step 2), Flux has already attempted an install. Check the HelmRelease status:
   - If READY=True: Flux may already own the resources. Verify by checking field managers on the Deployment. If Helm owns them, skip deletion and proceed to verification.
   - If error state: Delete the Helm release Secret first (`kubectl delete secret -n jellyfin -l owner=helm,name=jellyfin --kubeconfig ~/.kube/config-pi-k3s`), then proceed with resource deletion below.

B. Delete kubectl-managed Jellyfin resources (PRESERVE THE PVC):
   ```
   kubectl delete deployment jellyfin -n jellyfin --kubeconfig ~/.kube/config-pi-k3s
   kubectl delete service jellyfin -n jellyfin --kubeconfig ~/.kube/config-pi-k3s
   kubectl delete ingressroute jellyfin -n jellyfin --kubeconfig ~/.kube/config-pi-k3s
   ```
   DO NOT delete the PVC `jellyfin-config`. It holds Jellyfin's library database.

C. Force immediate Flux reconciliation:
   ```
   flux reconcile helmrelease jellyfin -n flux-system --kubeconfig ~/.kube/config-pi-k3s
   ```

D. Wait for HelmRelease to become READY (poll every 10 seconds, up to 3 minutes):
   ```
   flux get hr jellyfin -n flux-system --kubeconfig ~/.kube/config-pi-k3s
   ```
   If it does not become READY within 3 minutes, check Helm controller logs:
   ```
   kubectl logs -n flux-system deploy/helm-controller --tail=50 --kubeconfig ~/.kube/config-pi-k3s
   ```

E. If the install fails with field manager or "release already exists" errors, apply the fallback: add `force: true` under `spec.upgrade` in `flux/apps/jellyfin.yaml`:
   ```yaml
   spec:
     upgrade:
       force: true
       cleanupOnFail: true
   ```
   Commit and push, then reconcile again. Remove `force: true` after successful transfer.

F. Verify Jellyfin is running:
   ```
   kubectl get pods -n jellyfin --kubeconfig ~/.kube/config-pi-k3s
   kubectl get pvc -n jellyfin --kubeconfig ~/.kube/config-pi-k3s
   ```
   Verify PVC `jellyfin-config` still exists and is Bound.
