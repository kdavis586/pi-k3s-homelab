## Summary

`flux bootstrap github` is a single idempotent command that: (1) installs Flux controllers into the cluster, (2) generates and commits gotk-components.yaml + gotk-sync.yaml + kustomization.yaml to the repo, and (3) registers an SSH deploy key on GitHub. It is safe to re-run — it only commits if files have diverged. Flux v2.4.0 (already pinned in `all.yaml`) supports ARM64 and Kubernetes 1.29–1.31 which covers K3s's current default release track.

There are two bugs in the current Makefile `bootstrap-flux` target that must be fixed before running. First, `--path=k8s` is wrong — bootstrap creates a `flux-system/` subdirectory inside `--path`, so the correct value is `--path=flux` to produce `flux/flux-system/` which is where Phase 1 artifacts live. Second, the stub `k8s/flux-system/gotk-sync.yaml` uses HTTPS URL and points `path: ./k8s` — it will not conflict with bootstrap directly (different path tree), but it represents stale state and should be removed so the `k8s/` tree is not watched by any Flux Kustomization after Phase 2.

Once bootstrap runs with `--path=flux`, the generated `flux/flux-system/kustomization.yaml` will include all files in that directory, picking up the `apps-kustomization.yaml` committed in Phase 1. Flux will then reconcile both the Jellyfin and Pi-hole HelmReleases. Because `prune: false` is already set in `apps-kustomization.yaml`, existing workloads will not be deleted on first reconcile.

**Primary recommendation:** Fix `--path=k8s` → `--path=flux` in Makefile, delete `k8s/flux-system/gotk-sync.yaml`, run `make bootstrap-flux` with a fine-grained PAT.

## Architecture Patterns

### How Bootstrap Works

Bootstrap is a one-shot command that performs three actions in sequence:

1. **Installs controllers** into `flux-system` namespace on the cluster
2. **Commits manifests** to the git repo at `<--path>/flux-system/`:
   - `gotk-components.yaml` — all Flux CRDs and controller Deployments
   - `gotk-sync.yaml` — GitRepository + flux-system Kustomization CRDs
   - `kustomization.yaml` — plain kustomize file referencing the above two
3. **Registers SSH deploy key** on GitHub for the repository (read-only by default)

### Critical: --path Placement Rule

Bootstrap creates a `flux-system/` subdirectory **inside** whatever `--path` specifies. The directory structure produced is:

```
<--path>/
└── flux-system/
    ├── gotk-components.yaml    ← Flux controller manifests
    ├── gotk-sync.yaml          ← GitRepository + Kustomization CRDs
    └── kustomization.yaml      ← Plain kustomize config
```

For this repo, Phase 1 committed artifacts to `flux/flux-system/`. Therefore:
- Correct: `--path=flux` → creates `flux/flux-system/`
- Wrong (current Makefile): `--path=k8s` → would create `k8s/flux-system/`, conflicting with the stale stub

### Correct Repo Structure After Bootstrap

```
flux/
├── apps/
│   ├── jellyfin.yaml           ← HelmRelease (Phase 1)
│   ├── pihole.yaml             ← HelmRelease (Phase 1)
│   └── kustomization.yaml      ← plain kustomize config (Phase 1)
└── flux-system/
    ├── apps-kustomization.yaml ← Kustomization CRD for apps/ (Phase 1)
    ├── gotk-components.yaml    ← generated by bootstrap
    ├── gotk-sync.yaml          ← generated by bootstrap
    └── kustomization.yaml      ← generated by bootstrap
```

The bootstrap-generated `kustomization.yaml` in `flux/flux-system/` includes all YAML files in that directory, so `apps-kustomization.yaml` is automatically picked up without any extra configuration.

### Reconciliation Flow After Bootstrap

```
GitRepository (flux-system) polls github.com/kdavis586/pi-k3s-homelab@main [1m]
  └── Kustomization: flux-system  → applies flux/flux-system/
       └── Kustomization: apps    → applies flux/apps/
            ├── HelmRelease: jellyfin → chart: charts/jellyfin
            └── HelmRelease: pihole  → chart: charts/pihole
```

### gotk-sync.yaml Generated Content (expected)

Bootstrap generates a `gotk-sync.yaml` that looks like:

```yaml
# Source: generated by flux bootstrap github
apiVersion: source.toolkit.fluxcd.io/v1
kind: GitRepository
metadata:
  name: flux-system
  namespace: flux-system
spec:
  interval: 1m0s
  ref:
    branch: main
  secretRef:
    name: flux-system        # SSH key secret
  url: ssh://git@github.com/kdavis586/pi-k3s-homelab
---
apiVersion: kustomize.toolkit.fluxcd.io/v1
kind: Kustomization
metadata:
  name: flux-system
  namespace: flux-system
spec:
  interval: 10m0s
  path: ./flux/flux-system
  prune: true
  sourceRef:
    kind: GitRepository
    name: flux-system
```

Note: Bootstrap uses SSH URL (`ssh://git@github.com/...`) not HTTPS. The `secretRef` references the deploy key secret auto-created in `flux-system` namespace.

### Anti-Patterns to Avoid

- **Wrong `--path` value:** Using `--path=flux/flux-system` would create `flux/flux-system/flux-system/` — a double-nested path that breaks the layout
- **Running bootstrap before `prune: false` is set:** The apps Kustomization would default to `prune: true` and delete live workloads — but `prune: false` is already committed from Phase 1, so this is safe
- **Keeping the stale `k8s/flux-system/gotk-sync.yaml`:** This file is not in a watched path (since bootstrap watches `flux/`, not `k8s/`), but it is dead code that confuses future maintainers and should be deleted

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| SSH key generation + GitHub deploy key registration | Custom script using `ssh-keygen` + GitHub API | `flux bootstrap github` | Bootstrap handles both atomically; key is stored as K8s Secret in flux-system namespace automatically |
| Controller installation | `kubectl apply` of manually downloaded manifests | `flux bootstrap github` | Bootstrap pins versions, generates correct gotk-components.yaml, handles upgrades idempotently |
| Flux version pinning | Manually tracking releases | `flux_version: v2.4.0` in `all.yaml` | Already done; `--version` flag passes this to bootstrap |

**Key insight:** Bootstrap is designed to be the single entry point. Any work it does should not be replicated manually.

## Common Pitfalls

### Pitfall 1: Wrong --path Value in Makefile

**What goes wrong:** Current Makefile has `--path=k8s`. Bootstrap would create `k8s/flux-system/` instead of `flux/flux-system/`. The Flux-system Kustomization would watch `./k8s/flux-system` and miss the apps Kustomization in `flux/flux-system/apps-kustomization.yaml`.

**Why it happens:** The Makefile was scaffolded before the Phase 1 decision to use `flux/` as the root directory.

**How to avoid:** Change `--path=k8s` to `--path=flux` in the Makefile before running bootstrap.

**Warning signs:** After bootstrap, `flux get kustomizations -n flux-system` shows `flux-system` but NOT `apps`.

### Pitfall 2: Stale `k8s/flux-system/gotk-sync.yaml`

**What goes wrong:** The stub file committed before Phase 1 uses HTTPS URL and `path: ./k8s`. If it is ever applied (e.g., via `make deploy`) it creates a GitRepository pointing to the wrong path. It also creates a `flux-system` named Kustomization that conflicts with the one bootstrap creates.

**Why it happens:** The file was a placeholder from before Phase 1 planning.

**How to avoid:** Delete `k8s/flux-system/gotk-sync.yaml` before or as part of the bootstrap plan.

**Warning signs:** `flux get all` shows two Kustomizations both named `flux-system`.

### Pitfall 3: GITHUB_TOKEN with Insufficient Permissions

**What goes wrong:** Bootstrap fails with "403 Forbidden" when trying to register the deploy key on GitHub.

**Why it happens:** Classic PAT with only `read:user` or fine-grained PAT missing Administration permission.

**How to avoid:** Fine-grained PAT requires: Administration (Read/Write), Contents (Read/Write), Metadata (Read-only, mandatory). Classic PAT requires `repo` scope.

**Warning signs:** Bootstrap fails during the "creating deploy key" step, not the cluster apply step.

### Pitfall 4: flux CLI Not Installed on Operator Machine

**What goes wrong:** `make bootstrap-flux` fails with "flux CLI not found."

**Why it happens:** `flux` is not in PATH on the machine running `make`.

**How to avoid:** Makefile already has the guard check (`command -v flux`). Install via `brew install fluxcd/tap/flux` first.

**Warning signs:** Error message "Error: flux CLI not found. Run: brew install fluxcd/tap/flux" — this is the existing Makefile guard.

### Pitfall 5: Flux Version Mismatch Between CLI and all.yaml

**What goes wrong:** `flux bootstrap github` uses whatever version the CLI is, potentially different from `flux_version: v2.4.0` in `all.yaml`.

**Why it happens:** Homebrew installs latest; `all.yaml` pins v2.4.0.

**How to avoid:** Pass `--version=$(FLUX_VERSION)` to the bootstrap command, reading from `all.yaml` or a Makefile variable. This ensures the controllers installed match the intended version.

**Warning signs:** `flux get all` shows controller versions different from v2.4.0 after bootstrap.

## Code Examples

### Bootstrap Command (corrected)

```bash
# Source: flux bootstrap github docs + this repo's conventions
flux bootstrap github \
  --owner=kdavis586 \
  --repository=pi-k3s-homelab \
  --branch=main \
  --path=flux \
  --personal \
  --kubeconfig=$(KUBECONFIG)
```

Note: No `--token-auth` flag means SSH deploy key mode (default). The `GITHUB_TOKEN` env var is only needed at bootstrap time to register the key — after that, Flux uses the SSH key.

### Makefile Target (corrected)

```makefile
bootstrap-flux: ## Bootstrap Flux CD onto the cluster (requires GITHUB_TOKEN env var, flux CLI)
	@command -v flux >/dev/null 2>&1 || { echo "Error: flux CLI not found. Run: brew install fluxcd/tap/flux"; exit 1; }
	@[ -n "$(GITHUB_TOKEN)" ] || { echo "Error: GITHUB_TOKEN is not set"; exit 1; }
	flux bootstrap github \
		--owner=kdavis586 \
		--repository=pi-k3s-homelab \
		--branch=main \
		--path=flux \
		--personal \
		--kubeconfig=$(KUBECONFIG)
```

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| `flux bootstrap` with HTTPS + token stored in cluster | SSH deploy key (default) | Flux v2 design | More secure: token only needed at bootstrap, key is read-only |
| `gotk-sync.yaml` with HTTPS URL | SSH URL `ssh://git@github.com/...` | Flux v2 default | The stub in `k8s/flux-system/` uses HTTPS — this is why it must be replaced/deleted |
| `flux.toolkit.fluxcd.io/v1beta2` HelmRelease API | `helm.toolkit.fluxcd.io/v2` | Flux v2.3+ | Phase 1 correctly uses `helm.toolkit.fluxcd.io/v2` |
| Separate GitOps repo | Same-repo approach | Project decision | `flux/` tree co-located with `ansible/`, `charts/` etc. |

## Open Questions

1. **Flux version pinning in bootstrap command**
   - What we know: `flux_version: v2.4.0` is in `all.yaml`; Homebrew installs latest CLI
   - What's unclear: Whether the Makefile should pass `--version=v2.4.0` explicitly or rely on CLI version
   - Recommendation: Add `--version` flag to bootstrap command reading from a Makefile variable derived from `all.yaml`, or document that CLI must be pinned to v2.4.0

2. **Whether `k8s/flux-system/` directory should be fully removed or just `gotk-sync.yaml`**
   - What we know: `k8s/flux-system/` contains only `gotk-sync.yaml`; the `k8s/` tree will remain through Phase 3 (raw manifests for jellyfin/pihole still there)
   - What's unclear: Whether `make deploy` would apply the stale `gotk-sync.yaml` before it's deleted
   - Recommendation: Delete `k8s/flux-system/gotk-sync.yaml` as part of this phase; leave `k8s/flux-system/` dir removal for Phase 4 cleanup
