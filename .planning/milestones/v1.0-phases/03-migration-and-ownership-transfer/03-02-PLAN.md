This task resolves the Pi-hole node label blocker and transfers Pi-hole ownership to Flux.

**Step 1: Resolve node labels if missing (from Plan 01 pre-flight findings)**

Read the 03-01-SUMMARY.md to check if the pre-flight assessment found missing node labels. If `homelab/node-group=workloads` is NOT present on apple-pi and pumpkin-pi:

Run `make install-k3s` to apply node labels idempotently. This is the CLAUDE.md-approved way to apply labels (not kubectl label directly).

Verify labels applied:
```
kubectl get nodes --show-labels --kubeconfig ~/.kube/config-pi-k3s | grep homelab
```
Both apple-pi and pumpkin-pi should show `homelab/node-group=workloads`.

If labels ARE already present (from pre-flight), skip this step.

**Step 2: Handle DNS service gap (from Plan 01 pre-flight findings)**

Read the 03-01-SUMMARY.md for the pihole-dns service status from pre-flight.

- If `pihole-dns` LoadBalancer Service does NOT exist on cluster: no action needed, proceed.
- If `pihole-dns` LoadBalancer Service DOES exist on cluster: Delete it explicitly before ownership transfer since the Pi-hole chart does not include it and it would become an orphan:
  ```
  kubectl delete service pihole-dns -n pihole --kubeconfig ~/.kube/config-pi-k3s
  ```
  This is safe because Pi-hole uses `hostNetwork: true`, so DNS port 53 is bound directly on the host IP. LAN clients pointing at node IPs (192.168.1.101, 192.168.1.102) will still reach Pi-hole's DNS.

**Step 3: Delete kubectl-managed Pi-hole resources**

```
kubectl delete daemonset pihole -n pihole --kubeconfig ~/.kube/config-pi-k3s
kubectl delete service pihole-web -n pihole --kubeconfig ~/.kube/config-pi-k3s
kubectl delete ingressroute pihole -n pihole --kubeconfig ~/.kube/config-pi-k3s
```

Note: Pi-hole has no PVC to preserve (stateless DNS/DHCP).

**Step 4: Force immediate Flux reconciliation**

```
flux reconcile helmrelease pihole -n flux-system --kubeconfig ~/.kube/config-pi-k3s
```

**Step 5: Wait for HelmRelease READY (poll every 10 seconds, up to 5 minutes)**

Pi-hole DaemonSet may take longer than Jellyfin Deployment since it schedules on multiple nodes.

```
flux get hr pihole -n flux-system --kubeconfig ~/.kube/config-pi-k3s
```

If not READY within 5 minutes, check:
```
kubectl logs -n flux-system deploy/helm-controller --tail=50 --kubeconfig ~/.kube/config-pi-k3s
kubectl get pods -n pihole --kubeconfig ~/.kube/config-pi-k3s
kubectl describe daemonset pihole -n pihole --kubeconfig ~/.kube/config-pi-k3s
```

**Step 6: Fallback if needed**

If install fails, add `force: true` under `spec.upgrade` in `flux/apps/pihole.yaml`:
```yaml
spec:
  upgrade:
    force: true
    cleanupOnFail: true
```
Commit and push, then reconcile again. Remove `force: true` after successful transfer.

**Step 7: Verify both HelmReleases are READY**

```
flux get helmreleases -A --kubeconfig ~/.kube/config-pi-k3s
```

Both jellyfin and pihole must show READY=True. This is the gate for enabling prune in the next task.
